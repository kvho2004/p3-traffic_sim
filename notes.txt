## 2) Deadlock Analysis
- Analyze the system to identify **potential deadlocks**.
- Provide:
  - Explanation of **where and why** deadlock can occur.
  - Description of **how** it happens in this system.
  - A **concrete solution** to prevent it.
  

Definitions:

Deadlock - threads are stuck forever, each waiting on the other

Session 7.2: Four Conditions for Deadlock
1. Mutual Exclusion - threads claim exclusive control of resources 
(e.g., which thread stores P1, P2, P3 units? F1, F2, F3 dpts, ship, 5 trucks)

    Potential scenario: F3 has two departments. 
    - dpt 1 may have enough P1 units, but dpt 2 does not have enough P2 units.
    - dpt 2 may have enough P2 units, but dpt 1 does not have enough P1 units.
    - Both dpts are stuck waiting or inefficient because they cannot access the total # units

2. Hold-and-wait: Thread holds resources while waiting for additional ones 

    Prevention Strategy: acquire all locks atomically (std::lock())

3. No Preemption: Resources cannot forcibly removed from threads holding them

    Prevention Strategy: Instead of blocking, try to acquire lock & release when you hold if you cannot get it


4. Circular Wait: 
    Possible scenario: inconsistent order/race conditions
    Thread A: lock(f1_mtx) → lock(f3_mtx)
    Thread B: lock(f3_mtx) → lock(f1_mtx)  ← deadlock risk

    Prevention Strategy: acquire locks in the same order across all threads
    Ex: (common.h) 
    LOCK ORDERING RULE
    *   1. f1_out.mtx
    *   2. f2_out.mtx
    *   3. f3_in.p1.mtx
    *   4. f3_in.p2.mtx
    *   5. log_mtx
    *
    * Never acquire a lower-numbered lock while holding a higher-numbered one.

Potential Risk Scenarios:

1. Trucks starved at F3 unload
    F3 P2 storage: 5/5 (full, filled by previous trucks)
    5 trucks all waiting on not_full at F3
    F3 dept consumes 1 P2 → 1 space opens
    pthread_cond_broadcast wakes ALL 5 trucks
    All 5 check: only 1 has space → 4 go back to sleep
    1 truck unloads, signals not_empty
    Cycle repeats — trucks pile up waiting

Solution-> using pthread_cond_broadcast and while loops instead of if loops

2. F3 dpt needs both P1 and P2
    F3 dept holds (implicitly) a "committed to produce" state
    Consumed 2 P1 already, now waiting for 1 P2
    Ship can't unload P1 because it already did
    But if lock both storages simultaneously, circular wait is possible

Solution-> Using locks at the appropriate order, NOT lock both at once: 
        F3 dpt locks P1 storage, take units, 
        unlock F3,
        lock P2 storage seperately

3. F2 slow, trucks, starved at F2 loading
    F2 produces 1 P2 every 200ms
    5 trucks each need 2 P2 to depart = 10 P2 needed simultaneously
    F2 max storage = 5
    Trucks will fight over the 5 slots, most trucks starved

4. Potential Circular Wait #1 : 
    The ship arrives at F3 holding F3's P1 input lock, waiting for space. 
    While an F3 department holds its own production lock, waiting for more P1. 
    Neither can proceed.

5. Potential Circular Wait #2:
    A truck waits to unload at F3's P2 storage 
    while an F3 department holds a lock waiting on the truck's P2 to arrive
